# Быстрый старт для дизайнера

Этот раздел — короткий практический вход в BRIDGE. Чтобы подготовить первый макет, **не нужно сначала читать всю спецификацию**. Пройдите один пример ниже, затем откройте [каталог примеров](../../examples/README.ru.md).

## Что должно получиться

Разработчик, AI-агент или инструмент открывает Figma и без созвона понимает:

1. какую страницу и состояние он видит;
2. какие фреймы являются адаптивами одной страницы;
3. какие элементы на них соответствуют друг другу;
4. куда ведут ссылки и что делают кнопки;
5. что является контентом, декором или экспортируемым asset.

## Один пример от начала до handoff

Представим два адаптива лендинга и модалку обратной связи.

### Было: макет приходится разгадывать

```text
Desktop
  Frame 42
    Group 18
      Heading
      Text
    Button
    Button copy

Mobile final 2
  Group 91
    Title mobile
    Text
    Button

Popup
  ...
```

По этому дереву нельзя надёжно понять:

- `Desktop` и `Mobile final 2` — одна страница или разные;
- `Heading` и `Title mobile` — один элемент или два;
- что делают две кнопки;
- относится ли `Popup` к одной из кнопок;
- почему во втором адаптиве пропала кнопка.

### Стало: тот же макет описывает себя

```text
Главная [page=home] [route=/] [bp=1440] [view=default]
  Первый экран [section=home-hero]
    hero-copy
      hero-title
      hero-subtitle
    hero-actions
      email-link [href=mailto:sales@example.com]
      contact-button [action=modal:contact-modal]

Главная [page=home] [route=/] [bp=375] [view=default]
  Первый экран [section=home-hero]
    hero-copy
      hero-title
      hero-subtitle
    hero-actions
      email-link [href=mailto:sales@example.com]
      contact-button [action=modal:contact-modal]

Contact Modal [modal=contact-modal]
  modal-content
    modal-title
    close-button
```

Теперь видно, что:

- `[page=home]`, `[view=default]` и одинаковые имена слоёв связывают два адаптива;
- `[bp=1440]` и `[bp=375]` различают их ширину;
- `[href=mailto:sales@example.com]` обозначает переход к известной внешней цели;
- `[action=modal:contact-modal]` указывает на существующую `[modal=contact-modal]`;
- `close-button` — instance контрола закрытия из UI Kit, поэтому его поведение берётся из метаданных компонента;
- расположение, размеры, Auto Layout и стили по-прежнему читаются из Figma, а не из тегов.

> В реальной Figma раскладка двух адаптивов может быть разной. Важно сохранить тот же логический набор элементов и тех же родителей.

## Сделайте это за 5 шагов

### 1. Назовите корневые фреймы

Минимум для каждого адаптива:

```text
Название [page=page-id] [bp=ширина] [view=default]
```

Добавьте `[route=/production-path]`, только если настоящий маршрут уже известен. Не выдумывайте route ради заполнения тега.

### 2. Сопоставьте адаптивы

Положите рядом desktop и mobile и проверьте слой за слоем:

- один логический элемент имеет одно имя;
- элемент остаётся у того же логического родителя;
- смысл текста и действия не меняются;
- новый wrapper появляется только по реальной причине.

Размеры, направление Auto Layout, отступы, порядок и видимость могут отличаться.

### 3. Приведите важные имена к стабильному виду

Используйте короткий English `kebab-case`:

```text
hero-title
product-grid
contact-button
```

Не добавляйте ширину или устройство:

```text
hero-title-mobile   // плохо
button-375          // плохо
```

### 4. Добавьте только смысл, которого нет в Figma

| Figma уже знает | Нужен BRIDGE-тег |
| --- | --- |
| тип слоя, компонент, вариант | страница и состояние: `[page]`, `[view]` |
| Auto Layout, gap, padding | маршрут и адаптив: `[route]`, `[bp]` |
| размер, позицию, constraints | ссылка или действие: `[href]`, `[action]` |
| fill, stroke, effect, mask | цель: `[modal]`, `[state]` |
| иерархию frames и groups | переносимый смысл: `[section]`, `[decor]`, `[asset]` |

Не кодируйте в имени `row`, `gap=16`, `absolute` или `text`, если это уже однозначно задано в Figma.

### 5. Проведите проверку без автора

Попросите другого человека за пять минут ответить по одному только макету:

- Что это за страница и состояние?
- Где её адаптивы?
- Что произойдёт по клику на каждый интерактивный элемент?
- Где находятся modal/state targets?
- Что нужно экспортировать, а что должно остаться редактируемым?

Если ответ приходится сообщать устно, этой информации пока не хватает в макете.
