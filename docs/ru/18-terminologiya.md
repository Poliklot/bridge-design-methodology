# Терминология BRIDGE

Этот словарь задаёт единый язык русской документации. Его задача — сделать правила понятными дизайнеру и при этом сохранить точность канонических тегов, полей и идентификаторов.

## Основное правило

В обычном тексте используйте русское название. Английский термин оставляйте только тогда, когда он является:

- названием продукта или функции, например Figma или Auto Layout;
- названием библиотеки в файле, например `UI Kit` или `Page Sections`;
- частью синтаксиса BRIDGE, например `[page]`, `[bp]` или `[asset]`;
- полем данных, идентификатором правила или фрагментом кода;
- канонической частью расшифровки BRIDGE.

Если точный английский термин нужен для поиска по интерфейсу или внешней документации, укажите его один раз в скобках после русского объяснения.

## Шесть принципов

| Каноническое название | В русском тексте | Что означает |
| --- | --- | --- |
| Breakpoints | адаптивы и контрольные ширины | Варианты одной страницы для заданных ширин. |
| Roles | назначение слоёв | Роль элемента в структуре и продукте. |
| Identity | стабильные идентификаторы | Способ узнать один логический элемент в разных адаптивах и состояниях. |
| Dependencies | связи и действия | Переходы, действия и существующие цели этих действий. |
| Geometry | геометрия | Размеры, положение, отступы и параметры текста. |
| Exceptions | исключения | Намеренные отклонения с объяснимой причиной. |

Английские названия нужны в таблице, потому что их первые буквы образуют слово BRIDGE. В заголовках и объяснениях используйте русские формулировки.

## Макет и структура

| Не используйте в обычном тексте | Используйте | Примечание |
| --- | --- | --- |
| design | макет или проектирование | «Дизайн» допустим как название дисциплины, но конкретный файл лучше называть макетом. |
| design source | источник макета | Например, файл Figma или другой редактор. |
| root frame | корневой фрейм | Главный фрейм страницы, адаптива, состояния или секции. |
| frame | фрейм | Общепринятое название объекта Figma. |
| instance | экземпляр компонента | При первом упоминании лучше использовать полную форму. |
| source component | исходный компонент | Компонент, от которого создан экземпляр. |
| wrapper | обёртка | Родительский слой, который выполняет структурную задачу. |
| layout | раскладка или компоновка | `Auto Layout` остаётся названием функции Figma. |
| clipping | обрезка по границам | При необходимости рядом можно указать свойство Clip content. |
| hidden layer | скрытый слой | Не путать со слоем, который намеренно скрыт только на отдельном адаптиве. |

## Адаптивы и варианты

| Не используйте в обычном тексте | Используйте | Примечание |
| --- | --- | --- |
| breakpoint | контрольная ширина | В BRIDGE значение `[bp]` обозначает ширину подготовленного адаптива. |
| responsive | адаптивный | Например, «адаптивная структура». |
| desktop / tablet / mobile | широкий / планшетный / мобильный адаптив | В идентификаторах слоёв эти слова использовать не следует. |
| view | состояние страницы | Тег `[view]` остаётся без перевода. |
| state | состояние интерфейса | Тег `[state]` остаётся без перевода. |
| locale | язык и регион | Тег или поле `locale` в данных не переводится. |
| theme | тема оформления | Например, светлая и тёмная темы. |
| data scenario | сценарий данных | Например, пустой список или очень длинный текст. |

## Передача и проверка

| Не используйте в обычном тексте | Используйте |
| --- | --- |
| handoff | передача макета |
| BRIDGE-ready | подготовленный по BRIDGE; соответствующий BRIDGE |
| preflight | проверка перед передачей |
| validator | валидатор или средство автоматической проверки |
| linter | автоматическая проверка; линтер — только в названии продукта |
| rule catalog | каталог правил |
| severity | уровень серьёзности |
| error / warning / info | ошибка / предупреждение / справочная заметка |
| automatic / heuristic / manual | автоматическая / эвристическая / ручная проверка |
| pipeline | процесс или последовательность этапов |
| roadmap / phase | план развития / этап |
| false positive | ложное срабатывание |

Значения `error`, `warning`, `info`, `automatic`, `heuristic` и `manual` сохраняются в JSON и идентификаторах, но не должны без необходимости попадать в русское объяснение.

## Интерактивность

| Не используйте в обычном тексте | Используйте |
| --- | --- |
| clickable element | интерактивный элемент |
| control | элемент управления |
| action | действие |
| target | цель действия |
| destination | адрес или цель перехода |
| modal | модальное окно |
| submit | отправка формы |
| reset | сброс формы |
| anchor | якорь |
| route | маршрут |
| production route | рабочий маршрут продукта |
| draft | черновик или черновой статус |
| placeholder | временное значение или заглушка |

В синтаксисе BRIDGE сохраняются исходные формы: `[action]`, `[modal]`, `[route]`, `[control]`, `submit:form-id`.

## Содержимое и визуальные слои

| Не используйте в обычном тексте | Используйте |
| --- | --- |
| content | содержимое или продуктовый контент |
| asset | экспортируемый ресурс |
| decor | декоративный слой |
| visual intent | назначение визуального слоя |
| fixed height | фиксированная высота |
| overflow | переполнение |
| overflow policy | правила обработки переполнения |
| native text | редактируемый текст |
| rasterized text | текст, превращённый в изображение |
| fallback | запасной вариант |

Теги `[asset]` и `[decor]` не переводятся. Они являются частью машинно-читаемой грамматики.

## Стиль русской документации

- Обращайтесь к читателю на «вы».
- Формулируйте правила через конкретное действие: «Добавьте», «Проверьте», «Не меняйте».
- Не смешивайте русский и английский внутри одного выражения, если английская часть не является кодом или названием.
- Не переводите программный синтаксис.
- При первом упоминании объясняйте внутренний термин простыми словами.
- Используйте «макет» для конкретного файла и «методология» для BRIDGE.
- Различайте передачу макета, его реализацию и экспорт ресурсов: это разные действия.
- Не называйте правило «лучшей практикой», если оно является обязательной частью контракта.

## Примеры

Плохо:

> Final clickable element без action target блокирует handoff.

Хорошо:

> Интерактивный элемент без указанного действия блокирует передачу макета. Если действие ссылается на модальное окно или состояние, соответствующая цель должна существовать в файле.

Плохо:

> Stable identity не должна зависеть от breakpoint.

Хорошо:

> Один логический элемент должен сохранять один идентификатор на всех адаптивах. Не добавляйте в имя ширину или тип устройства.
