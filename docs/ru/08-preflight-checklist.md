# Полная проверка перед передачей

Эта проверка предназначена для строгого ревью макета и настройки автоматического валидатора. Для обычной самостоятельной проверки используйте более короткий [чеклист дизайнера](17-cheklist-dlya-dizaynerov.md).

## Режимы проверки

- **Быстрая** — самостоятельная проверка дизайнером до обсуждения макета.
- **Полная** — ревью перед началом реализации.
- **Проверка адаптера** — сопоставление контракта с возможностями конкретной среды реализации.

Для каждого пункта можно указать способ проверки:

- **автоматический** — результат однозначно определяется по данным макета;
- **эвристический** — валидатор находит подозрительный случай, но решение подтверждает человек;
- **ручной** — требуется содержательная оценка человека.

## Условия, блокирующие передачу

Макет не соответствует BRIDGE, если:

- у важного элемента нет стабильного идентификатора;
- обязательный продуктовый смысл нельзя получить из Figma или BRIDGE-тега;
- в одной области сравнения повторяются идентификаторы;
- идентификатор зависит от ширины или устройства;
- финальный интерактивный элемент не имеет известной ссылки или действия;
- действие ссылается на отсутствующую цель;
- между адаптивами без объяснения меняется количество элементов, вложенность или их продуктовый смысл;
- стабильный декоративный или экспортируемый визуал пропадает на одном адаптиве;
- текст имеет фиксированную высоту без правила обработки переполнения;
- динамический текст зависит от ручных переносов строк;
- скрытый слой используется как единственный источник актуальных данных.

## Стабильные идентификаторы

- [ ] Каждый важный элемент имеет стабильное английское имя в `kebab-case` или обязательный BRIDGE-тег.
- [ ] В именах нет суффиксов ширины и устройства: `-375`, `-768`, `-mobile`, `-desktop`.
- [ ] В одной странице, состоянии и контрольной ширине нет повторяющихся идентификаторов.
- [ ] Один логический элемент использует один идентификатор во всех адаптивах.
- [ ] Один логический элемент сохраняет одного логического родителя.
- [ ] Один идентификатор не обозначает разные типы элементов.
- [ ] `[decor]` и `[asset]` рассматриваются как признаки назначения, а не как дополнительные идентификаторы.
- [ ] Секционный ключ выводится из `Page Sections` или задан через `[section=...]`.

## Структура Figma

- [ ] Связанные элементы собраны во фреймы, группы или компоненты с понятной функцией.
- [ ] Auto Layout, ограничения, обрезка, исходный компонент и позиционирование заданы средствами Figma.
- [ ] Названия слоёв не дублируют технические свойства.
- [ ] Каждая обёртка имеет структурную причину.
- [ ] Обёртка, существующая только на одном адаптиве, объяснена как намеренное исключение.
- [ ] Скрытые старые версии не входят в передаваемую структуру.

## Адаптивы

- [ ] Каждый корневой фрейм адаптива имеет `[bp=...]`.
- [ ] Контрольная ширина объявлена на корневом фрейме и не повторяется в дочерних идентификаторах.
- [ ] Важные элементы присутствуют на всех необходимых адаптивах.
- [ ] Количество элементов коллекции не меняется ради красивого заполнения сетки.
- [ ] Порядок может меняться только внутри того же родителя.
- [ ] Скрытый на отдельной ширине элемент сохраняет идентификатор и логическое место.
- [ ] Смысл текста не меняется между адаптивами.
- [ ] Назначение визуального слоя не меняется: слой с `[decor] [asset]` не становится обычным контентом на другой ширине.

## Интерактивность

- [ ] Известный переход использует `[href=...]`.
- [ ] Неизвестная черновая ссылка использует `[link]`, а не `[href=#]`.
- [ ] Известное изменение интерфейса использует `[action=...]`.
- [ ] Неизвестное черновое действие использует `[control]`.
- [ ] Необязательные значения `[link=...]` и `[control=...]` используются только при необходимости машинного идентификатора.
- [ ] Перед финальной передачей разрешены все черновые `[link]` и `[control]`, если проект не допускает иной режим.
- [ ] Каждое действие на модальное окно, форму или состояние имеет существующую цель.
- [ ] Для модального окна задано закрытие.
- [ ] Для полей формы есть подписи и состояния.

## Маршруты и состояния страниц

- [ ] Корневой фрейм имеет стабильные `[page=...]`, `[bp=...]` и `[view=...]`.
- [ ] `[route=...]` или `[route-pattern=...]` указан только для настоящего рабочего маршрута.
- [ ] Отсутствующий маршрут учитывается как задача на будущее, а не заменяется вымышленным адресом.
- [ ] Маршрут стабилен во всех адаптивах и состояниях одной страницы.
- [ ] Обычное, пустое, загружаемое и ошибочное состояния сравниваются как отдельные значения `[view]`.

## Высота и переполнение

- [ ] Текст не имеет фиксированной высоты без явного ожидаемого поведения.
- [ ] Динамический текст не зависит от принудительных переносов строк или `<br>`.
- [ ] Карточки и секции не обрезают содержимое случайно.
- [ ] Фиксированная высота имеет продуктовую или техническую причину.
- [ ] Для обрезаемой области задано ожидаемое поведение: скрытие, многоточие, прокрутка или другое правило.
- [ ] Макет проверен на длинном содержимом и локализации.

## Контент, декор и экспорт

- [ ] Контентные изображения имеют стабильные имена.
- [ ] Редактируемый текст не экспортируется изображением без причины.
- [ ] Сложный визуал, который нужно экспортировать целиком, помечен `[asset]`.
- [ ] Декоративный слой помечен `[decor]`.
- [ ] Декоративный слой исключён из семантики содержимого и дерева доступности.
- [ ] Текст внутри корневого `[decor]` или `[asset]` не сравнивается как продуктовый контент.
- [ ] Корневой идентификатор декоративного или экспортируемого визуала всё равно стабилен между адаптивами.

## Компоненты и состояния

- [ ] Итоговое содержимое экземпляра учитывает локальные переопределения.
- [ ] Исходный компонент и его вариант можно определить автоматически.
- [ ] Состояние компонента не подменяется отдельной страницей.
- [ ] Состояние страницы не спрятано в варианте отдельного компонента.
- [ ] Отвязанные копии компонентов имеют объяснимую причину.
- [ ] Темы оформления не меняют продуктовый текст.

## Доступность

- [ ] Интерактивный элемент имеет доступное название.
- [ ] Цвет не является единственным способом показать состояние.
- [ ] Контраст текста и элементов управления достаточен.
- [ ] Размер области взаимодействия подходит для сенсорного ввода.
- [ ] Декоративные слои не попадают в дерево доступности.
- [ ] Ошибки формы связаны с соответствующими полями.
- [ ] Предусмотрены состояния фокуса.

## Итог ревью

Передача разрешена, когда:

- нет блокирующих ошибок;
- каждое предупреждение исправлено или объяснено;
- ручные вопросы получили ответ;
- возможности выбранного адаптера сопоставлены с используемыми функциями;
- человек, не участвовавший в подготовке файла, понимает макет без дополнительных пояснений.

Формат машинного отчёта и идентификаторы проверок описаны в разделе [«Валидация и автопроверки»](11-validaciya-i-avtoproverki.md).
