# Правила дизайна

BRIDGE-макет должен быть понятен до переноса. В Figma это достигается не ручными техническими тегами, а правильно собранной структурой, стабильными именами и явными продуктовыми намерениями.

## 1. Не дублируй то, что уже есть в Figma

Дизайнер не пишет в названии слоя технические свойства, которые Figma уже отдаёт как данные:

- тип слоя: текст, картинка, вектор, component instance;
- Auto Layout: направление, gap, padding, alignment, wrap;
- иерархию фреймов, групп и компонентов;
- позиционирование, constraints, clip content, размеры;
- fills, strokes, effects, masks;
- исходный компонент, variants, component properties.

Эти данные читает адаптер. Ручной тег нужен только там, где Figma не знает продуктовый смысл.

## 2. Что дизайнер объявляет руками

В названии слоя объявляется только смысл переноса:

- страница, реальный production route когда он известен, адаптив и view;
- секционный контракт;
- `href` для известных ссылок, `[link]` для draft-ссылок без href;
- `action` для известных controls/buttons, `[control]` для draft-действий без action;
- поле формы;
- цель для модалки или состояния;
- декоративный визуал;
- экспортируемый цельный asset;
- политика высоты, overflow или явное исключение, когда это нельзя безопасно вывести автоматически.

## 3. Структуру делай средствами Figma

Связанные элементы должны быть собраны как настоящая структура в Figma:

- текстовый блок и кнопки лежат в осмысленном родителе;
- карточки лежат в общем списке или сетке;
- заголовок, описание и CTA секции находятся внутри секции;
- повторяющиеся элементы оформлены одинаково;
- случайные `Frame 53`, `Group 271`, `copy 2` не используются как часть готового handoff.

Если элементы должны адаптироваться вместе, они не должны быть свободно разбросанными соседями.

## 4. Картинка, декор и asset — разные намерения

Контентная картинка — часть смысла. Если её убрать, пользователь потеряет информацию. Такая картинка получает стабильное имя слоя.

Декор — оформление. Если его убрать, смысл страницы сохранится. Декор помечается boolean-тегом визуального намерения `[decor]`.

`[decor]` означает:

- слой декоративный и не является продуктовым контентом;
- он не должен попадать в accessibility tree и может быть `aria-hidden`;
- ему не нужны alt/content semantics;
- он всё равно сохраняет stable responsive identity и не имеет права исчезать между адаптивами.

Asset — визуал, который нужно экспортировать целиком и не пересобирать из внутренних слоёв. Asset помечается boolean-тегом policy `[asset]`.

`[asset]` означает:

- визуал нужно экспортировать или использовать цельно;
- его нельзя пересобирать из внутренних слоёв;
- root asset layer всё равно сохраняет stable responsive identity между адаптивами.

Если у слоя уже есть хорошее стабильное имя, предпочитай boolean-теги:

```text
product-photo
article-cover
author-avatar
snow-bg [decor]
hero-glow [decor]
promo-poster [asset]
lab-illustration [asset]
sneg [decor] [asset]
```

Старый формат со значением остаётся только fallback для плохих/default имён слоёв:

```text
Frame 182 [decor=snow-bg]
Group 91 [asset=promo-poster]
```

## 5. Root frame — это конкретный адаптив страницы или секции

Корневой frame должен иметь стабильные page/view/breakpoint данные. `[route=...]` добавляется только когда известен реальный production route:

```text
Главная страница [bp=1920] [view=default] [page=home] [route=/]
Главная страница [bp=375] [view=default] [page=home] [route=/]
```

Если route пока неизвестен, не выдумывай fake production URL:

```text
Contacts Page [bp=1440] [view=default] [page=contacts]
```

Адаптив — это та же страница под другую ширину, а не новая версия смысла или структуры.

## 6. Обрезка и overflow — это решение

Если frame является карточкой, viewport’ом, маской или ограниченной поверхностью, поведение переполнения должно быть осознанным.

Текст из CMS, админки, каталога или локализации не должен держаться на ручных переносах строк. Он должен переноситься по ширине контейнера и иметь понятное поведение при переполнении.

## 7. Native/editable элементы остаются native

Текст, кнопки, поля формы, простые фигуры и иконки не нужно превращать в картинки без причины. Экспортируемый asset нужен только для сложного визуала, который целевая реализация не должна или не может пересобирать.

## 8. Методология не зависит от платформы

BRIDGE не привязан к конкретному инструменту. Figma — основной design source, но контракт должен переноситься во frontend-фреймворк, статический HTML/CSS, визуальный редактор, мобильный UI или внутреннюю дизайн-систему без угадывания.
