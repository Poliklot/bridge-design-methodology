# Правила дизайна

BRIDGE-макет должен быть понятен до переноса. Слой не считается готовым только потому, что он визуально стоит “на месте”.

## 1. У каждого слоя есть роль

Для переноса каждый слой должен быть одним из трёх типов:

- **Flow** — участвует в потоке layout: Auto Layout, stack, row, column, grid или аналогичная модель.
- **Absolute** — намеренно вынесен из потока и позиционируется координатами.
- **Target** — объявленная цель интерактивности: модалка, секция, состояние, внешний URL.

Если слой не находится явно в потоке и не объявлен как absolute, перенос становится неоднозначным.

## 2. Контейнеры выражают структуру

Используй layout-контейнеры для смысловых групп:

- `section-body flex`
- `content flex`
- `hero flex`
- `hero-copy flex`
- `button-row flex`
- `cards-grid flex`
- `stats-row flex`

Не раскладывай связанные элементы свободными соседями, если они должны вести себя как группа.

## 3. Absolute — только для осознанных исключений

Absolute подходит для:

- фонов;
- декоративных элементов;
- overlay;
- сложных атомарных иллюстраций;
- масок / экспортируемых постеров;
- элементов, которые намеренно выходят за границы родителя.

Absolute-слои должны иметь явные теги: `[abs]`, `[asset]`, `[decor]`, `[shape]`, `[overlay]`.

## 4. Root frame — это canvas конкретного breakpoint’а

Корневой frame секции должен иметь известную ширину breakpoint’а и стабильное имя:

```text
Pricing Section [bp=1200]
Pricing Section [bp=320]
```

Root frame может быть fixed-size snapshot’ом. Но внутренний контент всё равно должен раскрывать layout intent.

## 5. Clip content — это решение, а не случайность

Если frame является ограниченной поверхностью, карточкой, viewport’ом или секцией, поведение overflow должно быть понятно.

Если элемент должен визуально выходить наружу, вынеси его в absolute и положи на тот уровень иерархии, где этот overflow действительно задуман.

## 6. Native/editable элементы остаются native

Текст, кнопки и простые фигуры лучше оставлять редактируемыми. Asset fallback нужен только для сложных визуалов, которые целевая реализация не сможет точно воспроизвести.

## 7. Методология не зависит от платформы

BRIDGE не привязан к конкретному инструменту. Адаптер может переносить BRIDGE-контракт во frontend-фреймворк, статический HTML/CSS, визуальный редактор, мобильный UI или внутреннюю дизайн-систему, но исходный контракт макета остаётся тем же.
