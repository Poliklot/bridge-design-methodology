# Адаптивы и точки перелома

## Явные корневые фреймы

Создавай соседние фреймы для каждой адаптивной версии:

```text
Главная страница [bp=1920] [view=default] [page=home] [route=/]
Главная страница [bp=1280] [view=default] [page=home] [route=/]
Главная страница [bp=768] [view=default] [page=home] [route=/]
Главная страница [bp=375] [view=default] [page=home] [route=/]
```

Инструмент может вывести точку перелома из ширины фрейма, но явный тег безопаснее для передачи макета и сравнения.

## Один набор элементов, разная раскладка

Адаптив — это тот же экран, разложенный под другую ширину. Это не второй экран и не новая структура.

Корневые фреймы одной страницы, состояния или секции должны сохранять:

- один и тот же набор важных элементов с теми же ключами;
- ту же вложенность: элемент остаётся внутри того же родителя;
- то же количество элементов в списках и карточках, если явно не описано, что список динамический;
- те же действия, цели и текст для одних и тех же элементов.

Адаптивы могут менять размеры, отступы, переносы, направление авторазметки Figma, порядок, видимость и ограничения. Они не должны незаметно создавать элементы, удалять элементы или переносить один и тот же элемент в другого родителя.

Порядок и видимость — это свойства конкретного адаптива:

- элементы могут меняться местами внутри одного и того же родителя;
- элемент может быть скрыт на одном адаптиве и виден на другом;
- скрыть элемент — не значит удалить его: тот же ключ всё равно должен быть в структуре;
- видимый элемент нельзя заменять отдельной копией только для мобильного или только для широкого экрана.

Запрещённый случай — не «элемент сейчас не виден». Запрещённый случай — «этот ключ есть в одном адаптиве, но совсем отсутствует в другом» или «этот ключ переехал в другого родителя».

Если количество элементов или вложенность должны измениться, это уже не обычный адаптив. Оформи это как состояние, вариант компонента, правило коллекции, вариант под платформу или явное исключение BRIDGE.

```text
// широкий экран
button-group
  primary-cta [link=primary-cta] [href=/pricing]
  secondary-cta [control=secondary-cta] [action=modal:contact-modal]

// узкий экран
button-group
  primary-cta [link=primary-cta] [href=/pricing]
  secondary-cta [control=secondary-cta] [action=modal:contact-modal]
```

В Figma направление и размеры группы могут измениться, но набор элементов и их вложенность остаются теми же.

## Списки с постраничной навигацией

В каталоге товаров, выдаче поиска, списке заказов и любом другом списке с постраничной навигацией количество элементов на странице — это часть правил данных, а не способ выровнять сетку.

Адаптив может менять:

- количество колонок в сетке;
- отступы между карточками;
- ширину и высоту карточек;
- переносы и порядок внутри того же родителя.

Адаптив не должен:

- добавлять товары только потому, что на новой ширине сетка выглядит ровнее;
- убирать товары только потому, что последняя строка получилась неполной;
- менять количество товаров на странице без явного правила списка.

Неполная последняя строка — нормальная ситуация. Если на странице каталога показано 10 товаров, то при сетке 5 в ряд получится два полных ряда, а при сетке 4 в ряд — два полных ряда и ещё два товара. Добавлять ещё 2 товара ради красоты нельзя: это уже меняет состав выдачи, постраничную навигацию и общее число страниц.

Плохо:

```text
// широкий экран
products [collection=products]
  product-card-1
  ...
  product-card-10

// узкий экран
products [collection=products]
  product-card-1
  ...
  product-card-12
```

Хорошо:

```text
// широкий экран: 10 товаров, 5 колонок
products [collection=products]
  product-card-1
  ...
  product-card-10

// узкий экран: те же 10 товаров, 4 колонки
products [collection=products]
  product-card-1
  ...
  product-card-10
```

Если продукт действительно должен показывать разное количество товаров на разных ширинах, это не скрытое свойство адаптива. Такое отличие нужно оформить как явное правило списка: сколько элементов показывать, как считать страницы, что происходит с текущей страницей и какие состояния нужны для пустого, загружающегося или ошибочного списка.

## Изменение обёрток требует причины

Не добавляй случайные обёртки между адаптивами. Структура элементов — часть контракта, поэтому лучше добавить одну и ту же осмысленную обёртку на все адаптивы и менять только её настройки в Figma.

Обёртка, которая существует только на одном адаптиве, — это исключение. Она допустима только если выражает настоящую потребность: группировку, список, сетку, область обрезки, область наложения или декора, смысловую область или группировку под конкретную платформу. Причину нужно указать явно.

Плохо:

```text
// узкий экран
mystery-wrapper
  hero-title
```

Хорошо:

```text
// узкий экран
hero-copy
  hero-title
```

## Смысл текста должен пережить адаптив

Адаптив может менять размер, переносы, порядок и видимость. Он не должен менять текст или структуру элементов.

## Сначала точные адаптивы, потом плавное поведение

Сначала переносится точное состояние каждой заявленной ширины. Только после этого можно добавлять промежуточные правила, плавное изменение размеров или поведение по ширине контейнера.
