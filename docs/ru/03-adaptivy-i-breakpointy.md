# Адаптивы и контрольные ширины

BRIDGE рассматривает адаптивы как разные раскладки одной страницы, а не как независимые макеты.

## Объявляйте корневые фреймы явно

Создайте отдельный корневой фрейм для каждой подготовленной ширины:

```text
Главная [page=home] [route=/] [bp=1920] [view=default]
Главная [page=home] [route=/] [bp=1280] [view=default]
Главная [page=home] [route=/] [bp=768] [view=default]
Главная [page=home] [route=/] [bp=375] [view=default]
```

Значения `[page]`, `[route]` и `[view]` совпадают. `[bp]` меняется, потому что обозначает контрольную ширину конкретного фрейма.

Не добавляйте ширину в идентификаторы дочерних элементов.

## Сохраняйте один набор элементов

Адаптивы могут отличаться:

- размерами и положением;
- направлением Auto Layout;
- количеством колонок;
- порядком элементов внутри одного родителя;
- видимостью отдельных элементов;
- размерами шрифта и автоматическими переносами строк.

При этом они должны сохранять:

- идентификаторы значимых элементов;
- продуктовый смысл текста;
- действия и их цели;
- логических родителей;
- состав коллекций;
- назначение декоративных и экспортируемых слоёв.

Если элемент скрыт на одной ширине, это не означает, что его можно удалить из сравниваемой структуры. Валидатор должен понимать, что это тот же элемент с изменённой видимостью.

## Не подгоняйте коллекции под сетку

Количество карточек не должно зависеть от числа колонок.

Плохо:

```text
// 4 колонки
product-card-1 ... product-card-8

// 3 колонки
product-card-1 ... product-card-6
```

Хорошо:

```text
// на обеих ширинах
product-card-1 ... product-card-8
```

Неполная последняя строка допустима. Если на разных ширинах действительно показываются разные страницы коллекции, это должно быть описано отдельным правилом пагинации или сценарием данных.

## Объясняйте различия в обёртках

В идеале один и тот же элемент сохраняет одного родителя. Если узкому адаптиву нужна дополнительная группировка, сначала попробуйте использовать одну осмысленную обёртку на всех ширинах и менять только её настройки.

Обёртка, которая существует только на одном адаптиве, считается исключением. Она допустима, если выражает реальную потребность: список, сетку, область обрезки, наложение, смысловую группу или ограничение конкретной платформы.

Плохо:

```text
// узкий экран
mystery-wrapper
  hero-title
```

Хорошо:

```text
// все адаптивы
hero-copy
  hero-title
```

## Сохраняйте смысл текста

Адаптив может менять размер, автоматические переносы, порядок и видимость. Он не должен менять текст, цену, юридическую формулировку или назначение действия.

Если мобильной версии действительно нужен другой текст, оформите это как отдельный продуктовый вариант или явное исключение, а не как обычный адаптив.

## Сначала переносите точные состояния

Сначала реализация должна воспроизвести каждую заявленную контрольную ширину. После этого адаптер может добавить промежуточные правила, плавное изменение размеров или поведение по ширине контейнера.
