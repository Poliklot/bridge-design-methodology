# Чеклист дизайнера перед передачей макета

Используйте этот короткий чеклист перед передачей макета разработчику, ИИ-агенту или автоматизированному инструменту. Его цель — убедиться, что файл можно понять без устных пояснений.

Для строгого ревью и настройки валидатора используйте [полную проверку перед передачей](08-preflight-checklist.md).

## 1. Страница описана как продуктовая единица

**Проверьте:** по корневому фрейму понятно, какую страницу, контрольную ширину и состояние он показывает.

```text
Название страницы [page=page-id] [bp=1440] [view=default]
```

- [ ] `[page=...]` содержит стабильный идентификатор страницы.
- [ ] `[bp=...]` содержит ширину подготовленного адаптива.
- [ ] `[view=...]` различает обычное, пустое, загружаемое, ошибочное и другие состояния страницы.
- [ ] `[route=...]` добавлен только тогда, когда известен настоящий рабочий маршрут.
- [ ] Во всех адаптивах одной страницы совпадают `[page]` и `[view]`; меняется только `[bp]`.

Не придумывайте временный адрес ради заполнения `[route]`. Пока маршрут неизвестен, отсутствие тега считается задачей на будущее, а не ошибкой синтаксиса.

Подробнее: [маршруты и состояния](15-marshruty-stranic-i-sostoyaniya.md), [адаптивы](03-adaptivy-i-breakpointy.md).

## 2. Адаптивы описывают одну и ту же страницу

**Проверьте:** широкий, планшетный и мобильный варианты сохраняют одну продуктовую правду.

- [ ] Важные элементы присутствуют на всех необходимых адаптивах.
- [ ] Один элемент сохраняет одно имя и одного логического родителя.
- [ ] Текст, цена, юридические формулировки и назначение действий не меняются незаметно.
- [ ] Количество карточек в списке не подгоняется под число колонок.
- [ ] Перестановка элементов происходит внутри того же родителя.
- [ ] Реальное отличие структуры оформлено как состояние, вариант, правило коллекции или явное исключение.

Адаптив меняет раскладку, но не должен молча менять смысл страницы.

Подробнее: [адаптивы и контрольные ширины](03-adaptivy-i-breakpointy.md), [оси вариативности](16-osi-variativnosti.md).

## 3. Секции определяются однозначно

**Проверьте:** крупные повторяемые секции вынесены в `Page Sections`, а обычные элементы интерфейса остаются в `UI Kit`.

- [ ] Секционные компоненты находятся на странице библиотеки `Page Sections`.
- [ ] Кнопки, поля, карточки и другие элементы интерфейса находятся в `UI Kit`.
- [ ] Экземпляр секционного компонента не дублирует `[section=...]`, если ключ можно получить из исходного компонента.
- [ ] Обычная секция, собранная фреймом, имеет явный `[section=...]`.
- [ ] Один визуально и структурно одинаковый блок сохраняет один секционный ключ при смене контента.

Подробнее: [компоненты и секции](14-komponenty-i-ui-kit.md).

## 4. Важные элементы имеют стабильные идентификаторы

**Проверьте:** один логический элемент одинаково называется во всех адаптивах и состояниях.

- [ ] Значимые имена написаны по-английски в `kebab-case`.
- [ ] Имена не содержат ширину или тип устройства: `hero-title`, а не `hero-title-mobile`.
- [ ] Повторяющиеся элементы имеют различимые ключи: `product-card-1`, `product-card-2`.
- [ ] В одной области сравнения нет двух элементов с одинаковым идентификатором.
- [ ] Один идентификатор не используется для элементов разных типов.
- [ ] Технические имена редактора вроде `Frame 42` не остаются на значимых слоях.

Подробнее: [имена слоёв и стабильные идентификаторы](02-imena-sloev-i-identichnost.md).

## 5. Технические свойства не продублированы тегами

Figma уже является источником правды для:

- типа слоя;
- Auto Layout;
- размеров, положения и ограничений;
- вложенности фреймов, групп и компонентов;
- обрезки, масок, заливок, обводок и эффектов;
- исходного компонента, вариантов и свойств компонента.

- [ ] Названия слоёв не повторяют эти свойства вручную.
- [ ] BRIDGE-теги добавляют только продуктовый смысл, которого нет в метаданных Figma.
- [ ] При расхождении между названием и устройством слоя макет не создаёт две конкурирующие версии правды.

Подробнее: [правила подготовки макета](01-pravila-dizajna.md), [грамматика тегов](13-grammatika-tegov.md).

## 6. Структура собрана средствами Figma

**Проверьте:** отношения между элементами выражены иерархией и настройками макета, а не угадываются по их расположению.

- [ ] Связанные элементы находятся внутри понятного общего родителя.
- [ ] Auto Layout используется там, где элементы должны образовывать поток, список или сетку.
- [ ] Абсолютное позиционирование используется намеренно.
- [ ] Каждая обёртка отвечает за реальную задачу: группировку, раскладку, обрезку, фон или область взаимодействия.
- [ ] Случайных цепочек из одноэлементных обёрток нет.
- [ ] Отличие структуры на одном адаптиве объяснено.

Подробнее: [осмысленные обёртки](06-politika-obertok.md).

## 7. Интерактивность описана явно

**Проверьте:** у каждого интерактивного элемента известен результат действия.

- [ ] Переход по адресу или к якорю задан через `[href=...]`.
- [ ] Изменение интерфейса задано через `[action=...]`.
- [ ] Временно неизвестная ссылка помечена `[link]`, а не `[href=#]`.
- [ ] Временно неизвестное действие помечено `[control]`.
- [ ] Каждое действие на модальное окно или состояние ссылается на существующую цель.
- [ ] У элемента, состоящего только из иконки, есть понятное доступное название.
- [ ] Перед финальной передачей не осталось неразрешённых `[link]` и `[control]`, если проект не допускает черновой статус.

Подробнее: [интерактивность и цели действий](05-interaktivnost-i-celi.md).

## 8. Формы готовы к реализации

- [ ] Поле имеет понятное имя и идентификатор данных.
- [ ] Подпись поля связана с самим полем.
- [ ] Показаны или описаны обязательное, необязательное, ошибочное, отключённое и заполненное состояния.
- [ ] Кнопка отправки ссылается на существующую форму.
- [ ] Для загрузки, ошибки и успешной отправки предусмотрены состояния.
- [ ] Тип поля определяется компонентом или явно указан там, где его нельзя вывести автоматически.

Форма должна описывать не только внешний вид, но и ожидаемое поведение.

## 9. Текст выдерживает реальные данные

- [ ] Динамический текст переносится автоматически.
- [ ] Ручной перенос строки не используется для подгонки композиции.
- [ ] Блок выдерживает длинные слова, имена, цены, адреса и локализацию.
- [ ] Фиксированная высота текста имеет продуктовую причину.
- [ ] Для обрезки, прокрутки или скрытия переполнения задано ожидаемое поведение.
- [ ] Текст, который должен оставаться редактируемым, не превращён в изображение.

Подробнее: [высота и переполнение](07-vysoty-i-overflow.md).

## 10. Контент, декор и экспорт различаются по смыслу

- [ ] Контентная картинка имеет стабильное имя и остаётся управляемым содержимым.
- [ ] Декоративный слой помечен `[decor]`.
- [ ] Сложный визуал, который нужно экспортировать целиком, помечен `[asset]`.
- [ ] При необходимости один слой может быть одновременно `[decor] [asset]`.
- [ ] Теги не заменяют стабильное имя корневого визуального слоя.
- [ ] Декоративный слой не исчезает на отдельном адаптиве без объяснения.
- [ ] Текст не экспортируется изображением без явной причины.

Подробнее: [правила подготовки макета](01-pravila-dizajna.md).

## 11. Компоненты имеют понятный источник правды

- [ ] Страница использует экземпляры компонентов, а не отвязанные копии без причины.
- [ ] Изменённое содержимое экземпляра учитывается как итоговое содержимое страницы.
- [ ] Состояния компонента оформлены вариантами или свойствами, а не набором скрытых слоёв.
- [ ] Компонент интерфейса не объявляется секцией только из-за своего размера.
- [ ] Локальное переопределение секционного смысла используется только при необходимости.

Подробнее: [компоненты, `UI Kit` и `Page Sections`](14-komponenty-i-ui-kit.md).

## 12. Альтернативные состояния не спрятаны в мусорных слоях

- [ ] Пустое состояние, загрузка и ошибка оформлены как явные состояния страницы.
- [ ] Открытое меню, выбранная вкладка и другие состояния интерфейса описаны компонентом, вариантом или `[state=...]`.
- [ ] Скрытые старые версии и черновики не входят в передаваемую структуру.
- [ ] Скрытый слой не используется как единственный источник актуального содержимого.
- [ ] Намеренно скрытый элемент адаптива сохраняет свой идентификатор и логическое место в структуре.

## 13. Макет проходит проверку без автора

Перед передачей попросите человека, не участвовавшего в подготовке файла, ответить:

- [ ] Что это за страница и состояние?
- [ ] Где находятся её адаптивы?
- [ ] Какие секции переиспользуются?
- [ ] Что делает каждый интерактивный элемент?
- [ ] Где находятся цели действий?
- [ ] Какой текст является динамическим?
- [ ] Что нужно экспортировать?
- [ ] Какие исключения сделаны намеренно?

Если хотя бы один ответ приходится сообщать устно, зафиксируйте недостающую информацию в макете.

Следующий шаг: [полная проверка перед передачей](08-preflight-checklist.md).
