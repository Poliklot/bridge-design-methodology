# Контракт данных BRIDGE (черновик)

> **Статус: черновик до версии 1.0.** Эта страница описывает пример структуры данных, а не стабильный программный интерфейс. У BRIDGE пока нет стабильной JSON Schema и гарантий совместимости формата. Поля могут измениться до версии 1.0.

Подготовленный по BRIDGE макет раскрывает достаточно информации, чтобы реализацию не приходилось строить на догадках. Теги и каталог правил уже можно использовать; ниже показано, как будущий адаптер мог бы упаковать те же данные.

Актуальные ссылки: [статус и план развития](12-roadmap-proekta.md) · [исходный каталог правил](../../validator/rules.json)

## Обязательная информация

- Корневые фреймы адаптивов с `[bp=...]`.
- Стабильные идентификаторы BRIDGE: понятные имена слоёв и обязательные теги.
- Идентификатор и расположение источника: ID узла Figma, файл, страница и путь слоя.
- Структурные свойства из Figma: иерархия, Auto Layout, ограничения, позиционирование и обрезка.
- Назначение элемента: текст, кнопка, ссылка, фигура, изображение, ресурс, декор, модальное окно, секция или состояние.
- Геометрия, текстовые метрики, порядок соседних элементов и видимость на каждом адаптиве.
- Идентичность контента между адаптивами.
- Намерение взаимодействия: известные `href` / `action` или черновые маркеры `[link]` / `[control]`.
- Существующие цели ссылок, модальных окон, якорей и состояний.
- Правила экспорта ресурсов, высота, переполнение и явно оформленные исключения.

## Пример структуры данных

Версии идентифицируют этот пример, но не создают гарантию совместимости.

```json
{
  "contractVersion": "0.1-draft",
  "methodologyVersion": "0.7.1",
  "rulesVersion": "0.2.1",
  "source": {
    "tool": "figma",
    "fileKey": "example-file-key",
    "pageName": "Главная"
  },
  "breakpoints": [1200, 320],
  "tree": {
    "1200": { "hero": ["hero-title", "contact-cta"] },
    "320": { "hero": ["hero-title", "contact-cta"] }
  },
  "elements": [
    {
      "bridgeIdentity": "hero-title",
      "sourceLocation": {
        "figmaNodeId": "401:72",
        "layerPath": ["Главная / 1200", "Первый экран", "hero-title"]
      },
      "type": "text",
      "content": {
        "text": "Запустите магазин за один день",
        "sameAcrossBreakpoints": true
      },
      "layout": { "mode": "flow" },
      "fields": {
        "1200": { "left": 72, "top": 128, "width": 612, "fontSize": 58 },
        "320": { "left": 20, "top": 88, "width": 280, "fontSize": 34 }
      }
    },
    {
      "bridgeIdentity": "contact-cta",
      "sourceLocation": {
        "figmaNodeId": "401:94",
        "layerPath": ["Главная / 1200", "Первый экран", "contact-cta"]
      },
      "type": "button",
      "content": { "text": "Связаться" },
      "interaction": {
        "type": "modal",
        "targetBridgeIdentity": "contact-modal"
      }
    }
  ],
  "targets": [
    {
      "bridgeIdentity": "contact-modal",
      "sourceLocation": {
        "figmaNodeId": "408:11",
        "layerPath": ["Главная", "Модальные окна", "contact-modal"]
      },
      "type": "modal"
    }
  ]
}
```

## Откуда берутся поля

| Поле | Ожидаемый источник |
| --- | --- |
| `contractVersion` | Адаптер, который формирует экспериментальную структуру данных |
| `methodologyVersion` | Версия BRIDGE, по которой подготовлен файл |
| `rulesVersion` | Версия из [каталога правил](../../validator/rules.json) |
| `bridgeIdentity` | Понятные имена слоёв и теги BRIDGE |
| `sourceLocation` | Метаданные узла Figma, а также контекст файла, страницы и слоя |
| Структура и адаптивные поля | Иерархия Figma, параметры раскладки и фреймы адаптивов |
| Контент, взаимодействия и цели | Содержимое слоёв, данные прототипа и явное намерение BRIDGE |

## Статус и совместимость

| Уровень | Статус | Совместимость сейчас |
| --- | --- | --- |
| Теги BRIDGE и каталог правил | Можно использовать | Фиксируйте версии методологии и правил; до 1.0 возможны изменения |
| Модель данных на этой странице | Эксперимент | Только пример; поля могут измениться без миграции |
| Стабильная JSON Schema и политика совместимости | Запланированы до 1.0 | Пока отсутствуют |

В будущем адаптер сможет использовать подобную структуру, чтобы создать одну базовую модель и детерминированно применить адаптивные различия. До публикации схемы интеграциям следует читать актуальный каталог напрямую, а любой сериализованный набор данных считать специфичным для конкретного адаптера.
